# omok 프로젝트 기술 컨텍스트

## 개요
- 프로젝트명: omok
- 한 줄 설명: 오목(Gomoku) 게임 개발
- 생성일: 2026-07-10

## 확정 요구사항 (2026-07-10 사용자 확정)
- 목표 기력: 최상급 — 사람이 3수 이상 핸디캡을 갖고 시작해도 실전에서 이길 수 없는 수준 (practically superhuman). 단, 15x15 렌주는 완전히 풀리지 않아 "수학적 100% 불패 증명"은 불가 — 목표는 "실전 불패" 수준.
- 규칙: 렌주룰(RIF 표준) — 흑의 3-3/4-4/장목 금수 적용
- 보드 크기: 표준 15x15
- 플랫폼: 완전 오프라인. 와이파이 없는 사무용 노트북(GPU 없음)에 스탠드얼론 설치/실행.
- 과거 실패 사례(중요): 이전 유사 프로젝트에서 "고급" 난이도를 만들었으나 실제로는 쉽게 패배 — 원인은 얕은 탐색+VCT/VCF 부재+평가함수 미비였을 가능성이 높음. 이번엔 "기본 난이도" 자체를 최상급으로 설계해야 함(난이도 이름만 바꾸는 방식 금지).

## 플랫폼 / 기술 스택 (확정)
- 엔진: dhbloo/rapfi 채택 (C++, GPL-3.0, Gomocup 렌주 부문 최상위권 오픈소스 엔진, VCF/VCT 완전탐색+PVS+알파베타+Zobrist TT+렌주 금수판정 내장). 자체 재구현 대신 검증된 엔진을 채택해 "약한 평가함수/얕은 탐색" 리스크를 제거.
- 빌드 대상: Windows x64 네이티브 실행파일, Piskvork/pbrain 프로토콜
- GUI: 기존 Piskvork 프로토콜 호환 GUI(예: Yixin Board) 우선 채택. 한국어 커스텀 UI가 필요해지면 Python(PyQt)+서브프로세스 통신으로 후속 검토.
- 패키징: 설치 프로그램 없이 엔진.exe+GUI.exe+설정+GPL 소스 동봉 폴더를 zip으로 배포(xcopy 방식). GPU 불필요, 완전 오프라인.
- 라이선스 유의: GPL-3.0은 배포 시 소스 제공 의무 — 본 프로젝트는 사용자 개인 단독 설치용이라 소스 동봉으로 간단히 충족.

## 현재 범위
1차 목표: Rapfi 엔진을 렌주 룰로 빌드/검증하고 기존 GUI와 묶어 오프라인 노트북에 설치 가능한 배포 패키지 완성. 커스텀 GUI/한국어화는 스트레치 목표.

## 2단계 로드맵 (RL 고도화) — 계획만 기록, 아직 착수 안 함 (2026-07-10 developer 세션 조사)

### 배경
1단계(Rapfi 빌드/패키징) 완료 후 "Claude Code가 개입해 강화학습을 진행"하는 방향이 사용자 승인됨. 착수 전 실현 가능성을 조사함.

### 조사 결과 (dhbloo/rapfi 저장소 실사)
- Rapfi는 별도 Trainer 서브프로젝트(dhbloo/pytorch-nnue-trainer, PyTorch 기반)로 NNUE 신경망 "학습 코드"를 자체 보유함. 단, 이는 **지도학습(supervised)** 파이프라인으로 외부에서 미리 준비/가공된 기보 데이터셋(예: 가공된 KataGo self-play 데이터 포맷)을 입력받아 훈련하는 구조이며, **self-play 데이터 생성 도구 자체는 포함돼 있지 않음**.
- Rapfi 엔진 소스(Rapfi/tuning/ 디렉터리: tuner.cpp, optimizer.cpp 등)에는 고전 패턴 평가함수 가중치를 라벨링된 포지션 데이터셋으로 최적화하는 SPSA류 튜너가 별도로 존재 — 신경망이 아닌 경량 CPU 작업.
- 결론: "Rapfi에 완비된 self-play+RL 파이프라인을 그대로 돌리기만 하면 된다"는 전제는 틀렸음 — 데이터 준비/변환/오케스트레이션을 직접 구성해야 하는 반제품 수준.

### GPU 없는 사무용 노트북에서의 현실성 평가 (결론: 전면적 자체 RL은 비현실적)
- 신경망(NNUE) 처음부터 재학습: Trainer는 GPU 전제 설계(accelerate 분산학습 지원 언급)이며 CPU 전용 학습은 수십~수백 배 느림. AlphaZero/KataGo류 유의미한 강화는 통상 수백만 판의 self-play 데이터 + GPU 다수일 학습이 필요 — CPU 노트북 단독으로는 데이터 생성 단계부터 수 주~수개월 이상이 소요되며 개선을 보장할 수도 없음. "일단 해보면 된다"는 근거 없는 낙관으로 판단, 기각.
- 고전 평가함수 파라미터 미세조정(SPSA 튜너): CPU만으로 가능한 경량 작업이나, 이미 Gomocup 상위권 검증된 파라미터를 바닥부터 다시 튜닝하는 것은 실익이 낮고 기존 최적값을 훼손할 리스크가 있음.

### 결정된 대안 (아직 미착수, 계획만)
1. **1순위(기본)**: Rapfi 공식 배포 Networks(사전학습 가중치)를 그대로 사용. 이미 "실전 불패" 목표 기력에 부합하는 것으로 판단 — 추가 학습 불필요.
2. **2단계를 그래도 진행한다면**, "새 RL 파이프라인을 처음부터 구축"이 아니라 범위를 좁혀 재정의:
   - Claude Code가 Rapfi 엔진 대 엔진 self-play 매치를 오프라인 노트북에서 직접 실행해 소량(수백~수천 판) 기보를 생성하고, Rapfi/tuning의 SPSA 튜너로 "이미 검증된 파라미터 근방에서의 미세 파인튜닝"만 수행 — 새 신경망을 처음부터 훈련하는 것이 아니라 기존 엔진의 로컬 미세조정에 한정.
   - 전면적 NNUE 재학습(자체 self-play 데이터로 새 신경망을 처음부터 훈련)은 GPU/장시간 인프라를 확보하기 전까지 명시적으로 보류.
3. 이 로드맵은 아직 착수하지 않음 — 1단계(빌드/패키징) 완료 및 사용자 재승인 후 재검토.
